# Mezza Risk Platform PRD

Status: Draft, Part 1 captured; Phase 1 slice started
Last updated: 2026-06-05

## 1. Product Vision

Mezza Risk Platform should become the operating system for restaurant group underwriting, case tracking, bank-data review, document monitoring, and lending decision workflows.

Codex and Claude can be used for research, analysis, feature planning, and shipping implementation work, but the product source of truth must remain inside the Mezza app:

- Data is stored in the app database.
- Client links are generated by the app.
- Analysis and outputs are displayed in the app.
- Documents and status history are attached to app entities.
- Chat threads should help build and reason, not become the permanent system of record.

## 2. Current Reference Product

Reference URL: `https://mezza-platform.vercel.app/#`

The current Mezza platform reference contains useful group dashboard, tracker, and analytics concepts that should be incorporated into the risk platform app, while adapting them to the actual underwriting and data ingestion workflow.

## 3. Core Entities

- Group: parent restaurant/operator group.
- Venue: individual restaurant/location within a group.
- Case: underwriting application for a venue or group.
- Region: UAE or USA.
- Document: uploaded document with expiry metadata and extraction output.
- Bank connection: Stripe Financial Connections link/session.
- Bank account: connected bank account returned by Stripe.
- Bank transaction: transaction row saved from Stripe.
- Lending recommendation: system-calculated recommended lending amount.
- Final lending amount: manually adjustable final approved lending amount.
- Disbursement: pilot and subsequent disbursement events.

## 4. Group Dashboard

### 4.1 Requirement

Each signed restaurant group should have a group-level dashboard similar to the reference platform.

### 4.2 Group Summary

The lending amount section should be renamed and optimized:

- Replace current lending amount wording with `Recommended Lending Amount`.
- Add a separate `Final Lending Amount` field next to it.
- Remove the lower `P1`, `P2`, and `Disbursement Tracker` section from this summary view for now.

### 4.3 Lending Logic

Company policy:

- Final ceiling is the maximum approved lending capacity.
- Pilot disbursement is 20% of the final ceiling.
- Remaining approved amount may be disbursed quarterly.
- Example: if final ceiling is AED/USD 100,000:
  - Pilot: 20,000
  - Remaining possible exposure: 80,000
  - Future quarterly disbursements can be up to the outstanding approved amount, depending on usage and risk review.

### 4.4 Manual Override

The final lending amount must be editable/customizable by an authorized Mezza user.

The platform should preserve both:

- system recommended amount
- final approved amount

## 7. Cases / Tracker Page

The existing risk platform `Cases` page should absorb the tracker concepts from the reference platform.

### 7.1 Required Dates

Each case should track:

- case submission date
- first response date from the risk team
- final verdict date from the risk team

### 7.2 Region Sections

Cases should be split into:

- UAE
- USA

This can be done as tabs, segmented controls, or grouped sections. The best UX should be chosen based on density and scanability.

### 7.3 Case Reference Format

Case references should be shortened.

Current long IDs / overly hyphenated refs are not desirable for daily operations.

Requirement:

- keep UUID internally
- show a short human-readable case reference externally
- example format to evaluate: `MZA-001`, `UAE-001`, `USA-001`, or group-derived prefixes

### 7.4 Collapsible Group Tracker

The tracker should be group-first:

- top row: group name
- expandable dropdown: venues within the group
- venue rows show venue-level analytics/status

This allows scanning the portfolio by restaurant group rather than only by individual venue.

## 8. Analytics Page

The analytics page from the reference platform should be embedded into the risk platform.

- portfolio-level score distribution
- group-level exposure
- region split: UAE vs USA

Additional analytics can be added as the data model matures.

## 10. Document Expiry Monitoring

Add a daily document expiry monitoring feature.

### 10.1 Purpose

Mezza should know which client documents are expiring soon and be able to request replacements before underwriting or compliance risk emerges.

### 10.2 Expiry List

Create a list similar to a Vercel deployments list:

- most recently/soonest expiring documents at the top
- client/group/venue attached
- document type
- expiry date
- days remaining
- status

### 10.3 Actions

For each expiring document, user can:

- request updated document
- upload replacement document
- mark no longer needed
- close item
- keep pending

### 10.4 Status Sections

Document expiry items should be organized into:

- pending
- requested
- replaced/uploaded
- closed
- no longer needed

### 10.5 Email Drafting

The app should help generate a mail/message requesting the updated document.

The output should be stored or available in the app, not only in chat.

## 15. ACH Payments Module

Goal: allow Mezza to debit a US venue's connected bank account through Stripe ACH Direct Debit, while keeping underwriting data, payment state, mandates, and audit trail inside the Mezza app.

### 15.1 Core Concept

Current app flow:

`Client connects bank -> Mezza receives balances / transactions / ownership`

ACH requires an additional payment authorization flow:

`Client authorizes bank debit mandate -> Mezza can initiate ACH debit -> Stripe processes payment over 2-4 business days`

Financial Connections data access is not automatically payment authorization.

### 15.2 Required Stripe Setup

- Enable ACH Direct Debit / US bank account in Stripe live mode.
- Confirm account is eligible for ACH debits.
- Confirm Financial Connections live mode remains enabled.
- Add `payment_method` permission to future Financial Connections sessions.
- Add payment-related webhooks:
  - `payment_intent.processing`
  - `payment_intent.succeeded`
  - `payment_intent.payment_failed`
  - `charge.dispute.created`
  - `mandate.updated`, if applicable

### 15.3 Required App Changes

Add `payment_method` to future Financial Connections permissions.

Current:

`balances`, `transactions`, `ownership`

Future:

`balances`, `transactions`, `ownership`, `payment_method`

Existing connected accounts may need to reconnect/relink to grant this new permission.

### 15.4 ACH Setup Flow

Add a case-level action:

`Create ACH Mandate / Enable ACH Debit`

Create either:

- `SetupIntent` if saving the bank account for future debits
- `PaymentIntent` if charging immediately

### 15.5 Mandate Acceptance Page

Add a client-facing page explaining:

- amount or payment terms
- debit timing
- merchant/legal entity collecting payment
- ACH/Nacha authorization language
- cancellation/revocation terms

Client must explicitly accept before Mezza can debit.

### 15.6 ACH Tables

Add `ach_mandates`:

- id
- case_id
- stripe_customer_id
- stripe_payment_method_id
- stripe_mandate_id
- status
- accepted_at
- revoked_at
- raw

Add `ach_payments`:

- id
- case_id
- amount
- currency
- stripe_payment_intent_id
- status
- failure_reason
- initiated_at
- settled_at
- raw

Add `ach_events`:

- id
- stripe_event_id
- type
- payment_id
- raw
- created_at

### 15.7 ACH UI

Add ACH section on case bank page:

- mandate status
- saved payment method
- last 4
- account holder if available
- current payment status
- initiate debit button
- payment history table

### 15.8 ACH Statuses

ACH statuses should include:

- mandate_pending
- mandate_active
- payment_created
- processing
- succeeded
- failed
- disputed
- cancelled

### 15.9 Risk Controls Before Debit

Before initiating ACH, app should check:

- current/available balance if available
- recent NSF-like transaction patterns
- account active status
- ownership/signatory match if available
- manual analyst approval flag

### 15.10 Audit Trail

Every ACH action should log:

- who initiated it
- amount
- case
- timestamp
- Stripe IDs
- status changes
- webhook events

### 15.11 Product Rule

Mezza must not allow silent debits from a data-only bank connection. Payment authorization must be explicit and separate.

### 15.12 ACH Build Phases

- Phase 1: ACH readiness and schema
- Phase 2: mandate/setup flow
- Phase 3: one-off ACH debit
- Phase 4: payment status dashboard and webhooks
- Phase 5: risk checks before debit
- Phase 6: recurring/automated repayment logic, if needed later
